# 编程规范

## 1. 功能拆分原则

### 1.1 文件拆分
- **单一职责**：每个文件应只包含一个功能模块或相关功能的集合
- **功能分离**：不同的功能必须拆分在不同的文件中
- **模块划分**：按照功能模块组织文件结构

### 1.2 目录结构
- 遵循项目现有的分层架构：`data`、`domain`、`ui`
- 每个功能模块应有自己的子目录
- 例如：学习功能放在 `ui/learning/` 目录下

### 1.3 类的职责
- 每个类应专注于单一功能
- 避免创建超大类（超过500行代码）
- 当类的职责过多时，应考虑拆分为多个类

## 2. 通用功能复用

### 2.1 工具类
- 通用的功能应放在 `utils/` 目录下
- 工具类应是静态的或包含静态方法
- 工具类命名应清晰，表明其功能

### 2.2 组件复用
- 可复用的UI组件应放在 `ui/components/` 目录下
- 组件应具有良好的可配置性
- 避免硬编码，使用参数化设计

### 2.3 扩展函数
- 合理使用Kotlin扩展函数提高代码可读性
- 扩展函数应放在相关的工具类中
- 避免过度使用扩展函数，保持代码清晰

## 3. 文件长度限制

### 3.1 代码行数
- 每个Kotlin文件的代码行数不应过长
- **建议上限**：单个文件不超过300-400行
- **强制上限**：单个文件不超过500行

### 3.2 拆分策略
- 当文件接近长度限制时，考虑以下拆分方式：
  1. 按功能拆分：将不同功能拆分为不同文件
  2. 按层次拆分：将UI逻辑和业务逻辑分离
  3. 按模块拆分：将相关功能组织为模块

### 3.3 代码组织
- 使用空行和注释分隔不同的逻辑块
- 保持函数长度适中，单个函数建议不超过50行
- 避免嵌套过深的代码结构

## 4. 命名规范

### 4.1 文件命名
- 使用大驼峰命名法（PascalCase）
- 文件名应反映文件内容和功能
- 例如：`LearningViewModel.kt`、`WordRepositoryImpl.kt`

### 4.2 类命名
- 使用大驼峰命名法（PascalCase）
- 类名应清晰表达其职责
- 避免使用缩写，除非是广泛认可的缩写

### 4.3 函数命名
- 使用小驼峰命名法（camelCase）
- 函数名应是动词或动词短语
- 例如：`loadWords()`、`calculateProgress()`

### 4.4 变量命名
- 使用小驼峰命名法（camelCase）
- 变量名应清晰表达其用途
- 避免使用单个字符作为变量名（循环变量除外）

## 5. 代码风格

### 5.1 缩进和空格
- 使用4个空格进行缩进
- 避免使用制表符
- 在操作符两侧添加空格：`a + b` 而非 `a+b`

### 5.2 空行
- 函数之间使用一个空行分隔
- 逻辑块之间使用空行分隔
- 文件末尾保留一个空行

### 5.3 括号
- 左括号与前一行代码在同一行
- 右括号单独占一行
- 示例：
  ```kotlin
  fun processData() {
      // 代码
  }
  ```

### 5.4 分号
- Kotlin不需要分号，除非在同一行有多个语句
- 避免使用分号

## 6. 注释规范

### 6.1 类注释
- 为每个类添加类注释，说明其职责和用途
- 使用KDoc格式
- 示例：
  ```kotlin
  /**
   * 学习ViewModel，负责管理学习相关的状态和逻辑
   */
  class LearningViewModel : ViewModel() {
      // 代码
  }
  ```

### 6.2 函数注释
- 为公共函数添加注释，说明参数和返回值
- 复杂的业务逻辑应添加注释
- 避免过度注释，代码本身应清晰易懂

### 6.3 代码注释
- 对复杂的算法或逻辑添加注释
- 注释应简洁明了，解释"为什么"而不是"是什么"
- 使用中文注释，保持与项目其他注释风格一致

## 7. 错误处理

### 7.1 异常处理
- 使用try-catch处理可能的异常
- 避免捕获所有异常，应捕获具体的异常类型
- 对异常进行适当的处理，而非简单忽略

### 7.2 空安全
- 使用Kotlin的空安全特性
- 避免使用`!!`操作符
- 合理使用`?.`和`?:`操作符

### 7.3 输入验证
- 对用户输入进行验证
- 对函数参数进行边界检查
- 提供清晰的错误提示

## 8. 性能优化

### 8.1 内存管理
- 避免内存泄漏
- 正确使用生命周期感知组件
- 及时释放资源

### 8.2 计算优化
- 避免重复计算
- 使用缓存存储计算结果
- 优化循环和递归操作

### 8.3 网络优化
- 实现网络请求的缓存
- 减少不必要的网络请求
- 使用合适的网络请求库

## 9. 测试规范

### 9.1 测试覆盖
- 为关键功能编写单元测试
- 测试应覆盖正常情况和边界情况
- 保持测试代码的简洁性

### 9.2 测试命名
- 测试类名应以`Test`结尾
- 测试方法名应清晰表达测试目的
- 示例：`testLoadWordsReturnsCorrectList()`

### 9.3 测试隔离
- 测试应相互隔离，避免依赖
- 使用mock对象模拟依赖
- 测试环境应与生产环境隔离

## 10. 版本控制

### 10.1 提交规范
- 提交信息应清晰明了
- 提交信息应包含：功能、修改内容、原因
- 示例："添加学习进度计算功能"

### 10.2 分支管理
- 使用Git Flow工作流
- 主分支用于发布，开发分支用于日常开发
- 特性分支用于开发新功能

### 10.3 代码审查
- 提交代码前进行自我审查
- 代码审查应关注功能正确性、代码风格和性能
- 及时处理审查反馈

## 11. 最佳实践

1. **遵循Kotlin风格指南**：参考官方Kotlin风格指南
2. **使用现代Android组件**：优先使用Jetpack组件
3. **保持代码简洁**：避免冗余代码和复杂逻辑
4. **注重代码可读性**：编写易于理解的代码
5. **定期重构**：优化代码结构和性能
6. **学习新技术**：关注Android开发的新趋势

## 12. 常见问题及解决方案

### 12.1 文件过长
- **问题**：单个文件代码行数超过限制
- **解决方案**：按功能拆分文件，将相关功能组织为模块

### 12.2 功能耦合
- **问题**：不同功能之间耦合度高
- **解决方案**：使用接口分离，采用依赖注入

### 12.3 代码重复
- **问题**：相同或相似的代码出现在多个地方
- **解决方案**：提取公共代码为工具类或扩展函数

### 12.4 命名不规范
- **问题**：变量、函数或类的命名不清晰
- **解决方案**：遵循命名规范，使用描述性的名称
